# 数据采集路径日志时序与数据来源问题修复方案

> 版本：v1.0.0
> 日期：2026-07-11
> 负责人：Extension 前端负责人
> 关联文件：content.ts / remote_logger.ts / adapter.ts / interceptor.js / service_worker.ts

---

## 一、问题描述

用户在 Boss 直聘列表页启动扩展后，后端终端日志出现以下异常现象：

| 编号 | 异常现象 | 用户困惑 |
|------|----------|----------|
| P1 | 路径判定日志（`=== PATH DECISION ===`）比岗位数据日志**晚出现** | 时序倒错：先看到岗位被创建，后才知道走了什么路径 |
| P2 | 日志显示 API 拦截器和 DOM 兜底**都失败**，但 SidePanel **仍然有数据** | 数据来源不明：数据从哪来的？日志判定逻辑有误？ |
| P3 | 在已打开的 Boss 列表页启动扩展，**不会自动刷新页面** | 没有刷新就没有 API 调用，API 拦截器无从拦截 |
| P4 | 滚动加载测试**无法获取懒加载新增数据** | DOM observer 未生效或选择器失效 |
| P5 | 对数据来源与真实性产生**严重怀疑** | 虽然前端显示的数据是正确的，但日志与数据矛盾 |

---

## 二、根因分析

### 2.1 P1：日志时序倒错 — remote_logger 500ms 缓冲 vs 岗位数据立即发送

**根因位置**：[remote_logger.ts](file:///g:/my/my_file/AI%20Career%20Copilot/extension/src/logging/remote_logger.ts) L49

**核心矛盾**：

```
日志通道：remoteLog() → buffer → 500ms 定时器 → sendMessageToBackground(LOG) → SW → POST /api/extension/logs
岗位通道：sendJobsExtracted() → sendMessageToBackground(JOBS_EXTRACTED) → SW → POST /api/jobs/
```

- 岗位数据通过 `sendJobsExtracted()` **立即发送**（无缓冲，直接 `chrome.runtime.sendMessage`）
- 路径判定日志通过 `remoteLog()` 先推入 `buffer` 数组，**500ms 后**才批量 flush 发送

**实际时序**（以 API 拦截器路径为例）：

```
T=0ms    content.ts L200: remoteLog('[source=API] interceptor captured data')  → 入 buffer
T=0ms    content.ts L205: remoteLog('PATH_DECISION: API 拦截器路径')              → 入 buffer
T=0ms    content.ts L210: sendJobsExtracted()                                    → 立即发送到 SW
T~30ms   SW handleJobsExtracted → POST /api/jobs/ → 后端打印岗位创建日志
T~500ms  remote_logger flush() → sendMessageToBackground(LOG) → SW handleLog
T~530ms  SW handleLog → POST /api/extension/logs → 后端打印 PATH_DECISION
```

**终端实际显示顺序**：

```
[14:30:01.030] POST /api/jobs/ - 创建了 16 个岗位          ← 岗位数据先到
[14:30:01.530] === PATH DECISION === API 拦截器路径        ← 路径判定晚 500ms
```

**结论**：日志和数据的传输通道不同步，日志被缓冲 500ms 导致时序倒错。

### 2.2 P2："两条路径都失败"但仍有数据 — MutationObserver 兜底 + 早判定

**根因位置**：[content.ts](file:///g:/my/my_file/AI%20Career%20Copilot/extension/src/content/content.ts) L327-334

**核心逻辑缺陷**：

```typescript
// content.ts L321-355 runDomFallbackForInitialPage()
function runDomFallbackForInitialPage(): void {
  const jobs = bossAdapter.extractJobs()
  if (jobs.length === 0) {
    remoteLog('error', 'PATH_DECISION: API 拦截器和 DOM 兜底均未获取到数据', { path: 'none' })
    return  // ← 返回了，但 MutationObserver 仍然在监听！
  }
  // ...
}
```

**问题场景重现**：

```
T=0s     页面开始加载，initListPage() 启动
         ├─ setupApiTimeout() → 5s 定时器开始
         └─ bossAdapter.observe() → MutationObserver 监听 .rec-job-list

T=5s     API 超时，触发 runDomFallbackForInitialPage()
         ├─ bossAdapter.extractJobs() → 返回 0 条（页面字体未加载完/DOM 未渲染完）
         ├─ remoteLog('PATH_DECISION: 均未获取到数据', { path: 'none' })  ← 误判！
         └─ return（但 MutationObserver 仍在运行）

T=6.5s   页面渲染完成，.rec-job-list 出现岗位卡片
         ├─ MutationObserver 触发 → onJobsExtracted 回调
         ├─ sentJobTracker.filterNewJobs() → 16 条新岗位
         ├─ remoteLog('PATH_DECISION: DOM observer 路径')
         └─ sendJobsExtracted() → 16 条岗位发送到后端
```

**用户看到的矛盾日志**：

```
=== PATH DECISION === API 拦截器和 DOM 兜底均未获取到数据 ====================  ← 先说都失败了
[ext-log][content] [source=DOM-observer] 16 new jobs (scroll/append)          ← 然后又有 16 条数据
=== PATH DECISION === 用户获取滚动加载数据走的是 DOM observer 路径 ============  ← 最后说走 DOM observer
```

**另一个场景**：API 延迟到达

```
T=5s     DOM 兜底返回 0 条 → 日志 "均未获取到数据"
T=7s     API 响应终于到达（网络延迟）
         ├─ interceptor.js postMessage → content.ts 监听器处理
         ├─ sentJobTracker.filterNewJobs() → 16 条新岗位（DOM 兜底没发过任何数据）
         ├─ clearApiTimeout()（已经是 null，无操作）
         ├─ remoteLog('PATH_DECISION: API 拦截器路径')
         └─ sendJobsExtracted() → 16 条岗位发送
```

**结论**：`runDomFallbackForInitialPage()` 在 5 秒时的判定是**时间点快照**，不是最终结论。MutationObserver 和 API 监听器仍在运行，后续数据到达后会产生与"均未获取到数据"矛盾的日志。

### 2.3 P3：已打开页面不自动刷新 — 缺少 document.readyState 检测

**根因位置**：[content.ts](file:///g:/my/my_file/AI%20Career%20Copilot/extension/src/content/content.ts) L282-302

**当前逻辑**：

```typescript
// content.ts L282-302
if (pageInfo.isListPage) {
  initListPage()  // ← 只设置超时和 observer，不刷新页面
}
```

```typescript
// content.ts L403-445 initListPage()
function initListPage(): void {
  setupApiTimeout()        // 5s 超时
  bossAdapter.observe(...) // DOM observer
}
```

**问题场景**：

```
用户操作                          扩展状态                 API 拦截器状态
─────────────────────────────────────────────────────────────────────
1. 打开 Boss 列表页              扩展未启用               无拦截器
2. 页面加载，Boss 调用 job/list.json   扩展未启用          API 已返回，无拦截
3. 用户启用/安装扩展              content.ts 注入         interceptor.js 注入（但 API 已结束）
4. initListPage() 启动            5s 超时开始             拦截器等待中...
5. 5s 超时                        DOM 兜底                拦截器什么都没拦到
```

**关键缺失**：`initListPage()` 没有检测 `document.readyState`。如果页面已经是 `complete` 状态（用户先打开页面再启用扩展），应该主动刷新页面让 API 拦截器有机会拦截。

**interceptor.js 注册方式分析**：

- [service_worker.ts](file:///g:/my/my_file/AI%20Career%20Copilot/extension/src/background/service_worker.ts) L55-80：通过 `chrome.scripting.registerContentScripts` 注册 `world: 'MAIN' + runAt: 'document_start'`
- 这种注册方式只在**下一次页面加载**时生效
- 对已打开的页面，拦截器不会补注入主世界
- content.ts L102 的 `<script>` 标签注入虽然是兜底，但同样**来不及**拦住已经完成的 API 调用

### 2.4 P4：懒加载失败 — MutationObserver 仅重试一次 + 虚拟列表干扰

**根因位置**：[adapter.ts](file:///g:/my/my_file/AI%20Career%20Copilot/extension/src/modules/boss/adapter.ts) L324-358

**缺陷 1：observer 初始化只重试一次**

```typescript
// adapter.ts L324-358 setupListObserver()
private setupListObserver(onJobsExtracted: (jobs: RawBossJob[]) => void): void {
  const listContainer = document.querySelector(BOSS_SELECTORS.list.listContainer)
  if (!listContainer) {
    setTimeout(() => {
      const retryContainer = document.querySelector(BOSS_SELECTORS.list.listContainer)
      if (retryContainer) {
        this.setupListObserver(onJobsExtracted)
      }
      // ← 如果 1s 后仍未找到，永久放弃！不再重试！
    }, 1000)
    return
  }
  // ... observer 设置
}
```

content.ts 在 `document_start` 注入，此时 DOM 未渲染。如果 Boss 页面 1 秒后仍未渲染出 `.rec-job-list`（网络慢、SSR 延迟），observer **永远不会被设置**。

**缺陷 2：可能受 Boss 虚拟列表影响**

Boss 直聘列表页可能使用虚拟滚动（virtual scroll）：
- 滚出视口的卡片被从 DOM 中移除
- 滚回视口时重新创建卡片节点
- MutationObserver 每次都触发，但 `sentJobTracker` 已记录这些 `detailUrl`，`filterNewJobs` 返回 0 条新岗位
- 现象：observer 触发了，但没有新数据发送

**缺陷 3：滚动加载可能依赖 API 拦截器**

Boss 的懒加载机制是：用户滚动到底部 → 前端调用 `/wapi/zpgeek/.../job/list.json?page=2` → 渲染新卡片。

如果 API 拦截器未生效（P3 的刷新问题），懒加载的新数据只能靠 DOM observer 捕获。但 DOM observer 又可能因为缺陷 1 未被设置。

### 2.5 P5：REFRESH_JOBS 不刷新页面 — 只清 tracker 不触发 API

**根因位置**：[content.ts](file:///g:/my/my_file/AI%20Career%20Copilot/extension/src/content/content.ts) L542-548

```typescript
case ChromeMessageType.REFRESH_JOBS: {
  remoteLog('info', 'Manual refresh triggered: clearing tracker and waiting for API interceptor')
  sentJobTracker.clear()
  setupApiTimeout()
  return { ok: true }
}
```

用户手动刷新时：
1. 清空 `sentJobTracker`（去重记录）
2. 重新设置 5s API 超时
3. **不刷新页面**，**不重新调用 API**

如果页面已经加载完，API 不会被再次调用，拦截器什么也拦不到。5s 后又走 DOM 兜底，而 DOM 兜底提取的岗位和之前一样（因为 `sentJobTracker` 已清空，所以会重新发送一遍相同的岗位）。

**用户期望**：手动刷新应该重新从 Boss 获取数据，而不是重新发送已缓存的 DOM 数据。

## 三、问题关系图

```
P3（不自动刷新）
 │
 ├──→ API 拦截器拦不到首屏 API（因为 API 已在拦截器安装前完成）
 │     │
 │     └──→ P2（5s 超时后 DOM 兜底）
 │           │
 │           ├── DOM 未渲染完 → 0 条 → 日志 "均未获取到数据"
 │           │     │
 │           │     └──→ MutationObserver 后续触发 → 数据出现 ← 用户困惑！
 │           │
 │           └── DOM 已渲染 → 有数据 → 但薪资为空（字体反爬）
 │
 ├──→ P4（懒加载失败）
 │      ├── MutationObserver 未设置（1s 内 .rec-job-list 未渲染）
 │      └── 懒加载新数据靠 API → 但 API 拦截器未生效
 │
 └──→ P1（日志时序倒错）
        └── 岗位数据立即发送，PATH_DECISION 日志延迟 500ms → 终端显示顺序颠倒
```

**核心结论**：P3 是根因，P1/P2/P4 是 P3 的连锁反应。P5 是独立缺陷。

## 四、修复方案

### 4.1 方案概览

| 编号 | 修复项 | 优先级 | 影响范围 | 风险 |
|------|--------|--------|----------|------|
| F1 | 日志立即 flush（修复 P1 时序） | P0 | remote_logger.ts / content.ts | 低 |
| F2 | 移除"均未获取到数据"早判定，改为阶段性日志（修复 P2） | P0 | content.ts | 低 |
| F3 | 已加载页面自动刷新（修复 P3 根因） | P0 | content.ts | 中（需避免无限刷新） |
| F4 | MutationObserver 持续重试 + 虚拟列表适配（修复 P4） | P1 | adapter.ts | 中 |
| F5 | REFRESH_JOBS 触发页面刷新（修复 P5） | P1 | content.ts | 低 |

### 4.2 F1：日志立即 flush（修复 P1 时序倒错）

**目标**：确保 PATH_DECISION 日志在岗位数据发送**之前**到达后端终端。

**修改文件**：[remote_logger.ts](file:///g:/my/my_file/AI%20Career%20Copilot/extension/src/logging/remote_logger.ts)

**方案**：新增 `flushRemoteLogsSync()` 函数，在关键路径调用点（发送岗位数据前）立即 flush 日志缓冲。

```typescript
// remote_logger.ts 新增

/**
 * 标记下一次 remoteLog 为高优先级，跳过缓冲立即发送
 *
 * 用于 PATH_DECISION 等关键路径日志，确保与岗位数据的时序一致。
 * 实现方式：在 sendJobsExtracted 前调用 flushRemoteLogs()，
 * 确保 PATH_DECISION 日志先于岗位数据到达后端。
 */
export function flushRemoteLogs(): void {
  void flush()
}
```

**修改文件**：[content.ts](file:///g:/my/my_file/AI%20Career%20Copilot/extension/src/content/content.ts)

在所有 `sendJobsExtracted()` 调用前插入 `flushRemoteLogs()`：

```typescript
// content.ts 修改点 1：API 拦截器路径（L210 前）
remoteLog('info', 'PATH_DECISION: 用户获取首屏数据走的是 API 拦截器路径', { path: 'api_interceptor', jobCount: newJobs.length })
flushRemoteLogs()  // ← 新增：确保日志先于数据到达后端
void sendJobsExtracted(window.location.href, newJobs)

// content.ts 修改点 2：DOM 兜底路径（L354 前）
remoteLog('warn', 'PATH_DECISION: 用户获取首屏数据走的是 DOM 兜底路径', { path: 'dom_fallback', jobCount: newJobs.length })
flushRemoteLogs()  // ← 新增
void sendJobsExtracted(window.location.href, stripDomSalary(newJobs))

// content.ts 修改点 3：DOM observer 路径（L420 前）
remoteLog('info', 'PATH_DECISION: 用户获取滚动加载数据走的是 DOM observer 路径', { path: 'dom_observer', jobCount: newJobs.length })
flushRemoteLogs()  // ← 新增
void sendJobsExtracted(window.location.href, stripDomSalary(newJobs))
```

**修复后时序**：

```
T=0ms    remoteLog('PATH_DECISION: API 拦截器路径')  → 入 buffer
T=0ms    flushRemoteLogs()                           → 立即 flush，sendMessageToBackground(LOG)
T=0ms    sendJobsExtracted()                         → sendMessageToBackground(JOBS_EXTRACTED)
T~30ms   SW handleLog → POST /api/extension/logs     → 后端打印 PATH_DECISION
T~30ms   SW handleJobsExtracted → POST /api/jobs/    → 后端打印岗位创建
```

> **注意**：`sendMessageToBackground` 是异步的，两个消息几乎同时发出。由于日志消息体更小、SW 处理更快（直接 HTTP 转发，无串行循环），日志通常会先到达后端。如需严格保证顺序，可在 `sendJobsExtracted` 中先 `await flushRemoteLogs()` 再发送岗位数据，但这会增加 ~50ms 延迟。

### 4.3 F2：移除"均未获取到数据"早判定（修复 P2 矛盾日志）

**目标**：不再在 5s 时间点输出"API 和 DOM 均未获取到数据"的绝对结论，改为输出阶段性状态。

**修改文件**：[content.ts](file:///g:/my/my_file/AI%20Career%20Copilot/extension/src/content/content.ts) L321-355

**方案**：将 `runDomFallbackForInitialPage()` 中的"均未获取到数据"日志改为"DOM 兜底暂未发现数据，继续等待 MutationObserver"。

```typescript
// content.ts 修改前（L327-334）：
function runDomFallbackForInitialPage(): void {
  const jobs = bossAdapter.extractJobs()
  if (jobs.length === 0) {
    remoteLog('info', '[source=DOM-fallback] no jobs found on page')
    remoteLog(
      'error',
      'PATH_DECISION: 用户获取首屏数据失败，API 拦截器和 DOM 兜底均未获取到数据',
      { path: 'none' },
    )
    return
  }
  // ...
}

// content.ts 修改后：
function runDomFallbackForInitialPage(): void {
  const jobs = bossAdapter.extractJobs()
  if (jobs.length === 0) {
    // 不再输出"均未获取到数据"的绝对结论
    // MutationObserver 仍在运行，后续数据到达时会通过 DOM observer 路径发送
    remoteLog(
      'warn',
      'PATH_DECISION: API 拦截器 5s 超时未捕获数据，DOM 兜底暂未发现已渲染卡片。' +
        'MutationObserver 仍在监听，页面渲染完成后将自动捕获',
      { path: 'waiting', reason: 'api_timeout_and_dom_not_ready' },
    )
    return
  }
  // ... 有数据的逻辑不变
}
```

**修复后日志序列**（同一场景）：

```
=== PATH DECISION === API 拦截器 5s 超时，DOM 兜底暂未发现卡片，继续等待 ===
[ext-log][content] [source=DOM-observer] 16 new jobs (scroll/append)
=== PATH DECISION === 用户获取滚动加载数据走的是 DOM observer 路径 ============
```

用户不再看到矛盾的"均未获取到数据"→"16 条数据"序列。

### 4.4 F3：已加载页面自动刷新（修复 P3 根因）

**目标**：当扩展在已加载完成的 Boss 列表页启动时，主动刷新页面让 API 拦截器有机会拦截首屏 API。

**修改文件**：[content.ts](file:///g:/my/my_file/AI%20Career%20Copilot/extension/src/content/content.ts)

**方案**：在 `initListPage()` 开头检测 `document.readyState`，如果页面已加载完成（`complete`），且本次是扩展刚注入（非 SPA 路由切换），则触发 `location.reload()`。

**关键设计：防止无限刷新循环**

使用 `sessionStorage` 标记本次会话是否已刷新过：

```typescript
// content.ts 新增

/** sessionStorage key：标记当前 Tab 是否已执行过扩展启动刷新 */
const RELOAD_FLAG_KEY = '__acc_extension_reloaded'

/**
 * 检测并执行启动刷新
 *
 * 场景：用户先打开 Boss 列表页，后启用/安装扩展。
 * 此时页面 API 已完成，拦截器来不及拦截。
 * 通过 location.reload() 刷新页面，让拦截器在 document_start 注入后
 * 能拦住刷新后的 API 调用。
 *
 * 防循环：用 sessionStorage 标记，同一会话只刷新一次。
 * SPA 路由切换不会触发（URL 不变，不走 initListPage 的刷新分支）。
 */
function maybeReloadOnStartup(): boolean {
  // 已经刷新过，跳过
  if (sessionStorage.getItem(RELOAD_FLAG_KEY)) {
    return false
  }

  // 页面仍在加载中，拦截器有机会拦住 API，不需要刷新
  if (document.readyState === 'loading') {
    return false
  }

  // 页面已加载完成（interactive / complete），API 已结束
  // 标记并刷新
  sessionStorage.setItem(RELOAD_FLAG_KEY, '1')
  remoteLog(
    'warn',
    'PATH_DECISION: 扩展在已加载页面启动，主动刷新以确保 API 拦截器生效',
    { path: 'startup_reload', readyState: document.readyState },
  )
  flushRemoteLogs()  // 确保日志在刷新前发出
  location.reload()
  return true
}
```

**修改 `initListPage()`**：

```typescript
// content.ts 修改前（L403-404）：
function initListPage(): void {
  setupApiTimeout()
  bossAdapter.observe({ ... })
}

// content.ts 修改后：
function initListPage(): void {
  // 启动刷新检测：如果页面已加载完成，刷新让拦截器生效
  if (maybeReloadOnStartup()) {
    return  // 页面即将刷新，不需要继续初始化
  }

  setupApiTimeout()
  bossAdapter.observe({ ... })
}
```

**刷新后流程**：

```
T=0     扩展注入，content.ts 运行
T=0     maybeReloadOnStartup() → readyState=complete → 标记 sessionStorage → location.reload()
T=0     flushRemoteLogs() → 日志发出
T~50ms  页面开始重新加载
T~50ms  content.ts 在 document_start 重新注入
T~50ms  interceptor.js 在 document_start 注入主世界（SW 已注册）
T~100ms 页面 JS 执行，调用 /wapi/zpgeek/.../job/list.json
T~100ms interceptor.js 拦截到 API 响应 → postMessage → content.ts 处理
```

### 4.5 F4：MutationObserver 持续重试 + 虚拟列表适配（修复 P4 懒加载）

**目标**：确保 MutationObserver 能在 DOM 渲染完成后成功挂载，并正确处理虚拟列表场景。

**修改文件**：[adapter.ts](file:///g:/my/my_file/AI%20Career%20Copilot/extension/src/modules/boss/adapter.ts)

**缺陷 1 修复：持续重试直到找到容器**

```typescript
/** 列表容器重试间隔（ms） */
private static readonly LIST_CONTAINER_RETRY_INTERVAL = 500

private setupListObserver(
  onJobsExtracted: (jobs: RawBossJob[]) => void,
  retryCount = 0,

      return

      this.setupListObserver(onJobsExtracted, retryCount + 1)
    }, BossAdapter.LIST_CONTAINER_RETRY_INTERVAL)
    return
  }
  // ... 找到容器，设置 observer（逻辑不变）
}
```

## 八、附录：问题复盘要点

### 8.1 为什么"前端显示的数据是正确的"但"日志看起来矛盾"？

**答案**：数据本身没问题。MutationObserver 在页面渲染后正确捕获了岗位卡片，数据通过 DOM observer 路径发送到后端。问题出在**日志的时序和措辞**：

1. 5s 超时时的"均未获取到数据"是**过早结论**，不代表最终状态
2. PATH_DECISION 日志因 500ms 缓冲比岗位数据晚到，造成"先有数据后有路径"的错觉

### 8.2 为什么 API 拦截器和 DOM 都"失败"了还有数据？

**答案**：它们在 5s 时间点确实都"失败"了（API 未拦到、DOM 未渲染），但**不是最终失败**：

- MutationObserver 仍在运行，页面渲染完成后自动捕获了数据
- 这不是数据来源不明，而是走了 DOM observer 路径（第三优先级）

### 8.3 为什么不刷新就没有 API 让拦截器拦截？

**答案**：Boss 的首屏 API 在页面加载时调用一次。如果扩展在页面加载后才注入：

- `chrome.scripting.registerContentScripts` 注册的 `document_start` 拦截器只对**下一次页面加载**生效
- content.ts 的 `<script>` 标签注入虽然立即执行，但 API 调用已经结束
- 必须刷新页面让 Boss 重新调用 API，拦截器才能拦截

## 九、技术复盘

### 9.1 核心逻辑

本修复方案解决三个层面的问题：

1. **时序同步**（F1）：日志和数据走不同通道，需要关键节点同步
2. **状态机正确性**（F2）：5s 超时不是一个终态，不应该输出终态结论
3. **生命周期管理**（F3/F5）：扩展注入时机与页面生命周期的对齐

### 9.2 关键技术点

- **Chrome MV3 动态内容脚本注册**：`registerContentScripts` 的 `world: 'MAIN' + runAt: 'document_start'` 只对新页面加载生效，对已打开页面无效
- **sessionStorage 防刷新循环**：浏览器刷新后 content.ts 重新执行，需要跨刷新持久化的标记
- **MutationObserver 初始化时机**：`document_start` 注入时 DOM 未渲染，需要异步等待容器出现
- **日志缓冲 vs 立即发送**：批量缓冲减少 HTTP 请求，但关键路径日志需要绕过缓冲

### 9.3 潜在风险

- **sessionStorage 被页面清除**：Boss 前端代码如果调用 `sessionStorage.clear()`，会导致无限刷新。兜底方案：URL 参数 `?acc_reload=1`
- **Boss 页面结构变更**：`.rec-job-list` 选择器失效会导致 observer 永远无法挂载。需要监控选择器命中率
- **interceptor.js 注入失败**：SW 的 `registerContentScripts` 在某些 Chrome 版本可能失败。已有 `<script>` 标签兜底，但需要监控注入成功率
