# API 拦截器失效与懒加载 401 修复方案

> 日期：2026-07-11
> 涉及模块：extension/public/interceptor.js、extension/src/content/content.ts、extension/src/background/router.ts、extension/src/background/backend_client.ts

---

## 一、问题整理

### 问题 P1：API 拦截器一直失败，首屏数据总是走 DOM 兜底

**现象**：终端日志始终显示 `PATH_DECISION: 用户获取首屏数据走的是 DOM 兜底路径`，从未走 API 拦截器路径。

### 问题 P2：拦截器是否需要刷新才能拦截到 API？

**用户疑问**：发现刷新是在获取到数据之后才发生的，不确定刷新是否影响拦截。

### 问题 P3：DOM 兜底触发太快，是 API 超时还是报错？

**现象**：日志中 `Content script started` 和 `interceptor did not capture data within 5s` 几乎同时出现（间隔 1ms），DOM 兜底立即提取到 15 个职位。

### 问题 P4：API 拦截器是否应该设置超时？

**用户观点**：拦截器应该是一直监听是否有请求响应数据的，不应该有超时。

### 问题 P5：懒加载新增数据报 401

**现象**：DOM 路径下获取懒加载的新增数据时，POST `/api/jobs/` 返回 401。

## 二、根因分析

### P1 根因：`event.source !== window` 错误过滤了主世界消息（已修复）

**数据流**：
```
Boss 页面 JS 调用 fetch/XHR
  ↓
interceptor.js（MAIN world）拦截到响应
  ↓
window.postMessage(...)    ← 主世界的 window
  ↓
Content Script（isolated world）的 message 监听器
  ↓
event.source !== window    ← false！主世界 window ≠ isolated world window
  ↓
return（消息被丢弃）
```

**原理**：Chrome 扩展 MV3 的多世界模型中，MAIN world 和 isolated world 各自有独立的 `window` JavaScript 对象。`event.source` 指向发送方（MAIN world）的 `window`，而 Content Script 里的 `window` 是 isolated world 的 `window`，两者永远不相等。

**修复**：改为 `event.origin !== window.location.origin`，只校验来源 origin，不比较 window 对象。

> 注：此修复已在上一轮完成。

### P1 补充：interceptor.js 的 TDZ bug（已修复）

`sendLog()` 在 `const LOG_MESSAGE_TYPE` 声明之前被调用，导致 `ReferenceError` 被 try/catch 静默吞掉。

**修复**：将所有 `const` 声明移到 `sendLog()` 首次调用之前。

> 注：此修复已在上一轮完成。

### P1 待确认：Boss API URL 是否匹配 TARGET_API_PATTERNS

当前匹配的 URL 模式：
```javascript
const TARGET_API_PATTERNS = [
  '/wapi/zpgeek/pc/recommend/job/list.json',
  '/wapi/zpgeek/search/job/list.json',
  '/wapi/zpgeek/job/list.json',
]
```

如果 Boss 直聘修改了 API 路径，拦截器不会匹配到。已在 interceptor.js 中加了 `DEBUG_ALL_REQUESTS = true`，会记录所有 fetch/XHR 请求的 URL。**测试时需要观察终端是否出现 `[fetch] intercepted request` 或 `[xhr] open intercepted` 日志**：

- 如果出现了这些调试日志 → 拦截器在工作，但 URL 不匹配 → 需要扩大 `TARGET_API_PATTERNS`
- 如果没出现 → 拦截器脚本未执行或 `sendLog` 仍失败 → 需要检查 SW 注册和 `<script>` 注入

### P2 根因：拦截器注入时机与刷新的关系

**拦截器的本质**：hook `window.fetch` 和 `XMLHttpRequest.prototype`。必须在 Boss 页面 JS 发起 API 请求**之前**安装，否则该请求拦不到。

**两条注入路径**：

| 路径 | 机制 | 时序 | 是否需要刷新 |
|------|------|------|-------------|
| 主路径 | SW 通过 `chrome.scripting.registerContentScripts` 注册 `world: 'MAIN'` + `runAt: 'document_start'` | Chrome 在页面任何 JS 之前同步注入，无竞态 | **不需要** |
| 兜底路径 | Content Script 通过 `<script>` 标签注入 | 标签插入 DOM 后异步加载，Boss JS 可能先执行 | **需要**（刷新后主路径生效） |

**结论**：
- 如果 SW 注册成功（主路径），**不需要刷新**。拦截器在 `document_start` 就位，能拦到首屏 API。
- 如果 SW 注册失败（走兜底路径），**需要刷新**。但刷新后如果主路径仍失败，还是拦不到。
- 当前代码**总是**走兜底路径（Content Script 无条件注入 `<script>` 标签），不管 SW 是否注册成功。如果 SW 注册成功，兜底注入会因 `__bossJobInterceptorInstalled` 标志跳过（无害）。
- `maybeReloadOnStartup()` 只在页面已加载完成（`readyState !== 'loading'`）时刷新，确保拦截器有机会拦截刷新后的 API。

### P3 根因：日志缓冲延迟导致时序错觉

**日志时序分析**：
```
12:41:59.618 | Content script started                              ← 当前轮次
12:41:59.619 | [injector] injecting via <script> tag fallback       ← 当前轮次
12:41:59.619 | [source=API] interceptor did not capture within 5s   ← 上一轮的延迟日志！
12:41:59.619 | [source=DOM-fallback] starting DOM fallback          ← 上一轮的延迟日志！
12:41:59.621 | [source=DOM-fallback] extracted 15 new jobs          ← 上一轮的延迟日志！
```

**原理**：`remoteLog` 有 500ms 的批量缓冲（`FLUSH_INTERVAL_MS = 500`）。后端终端看到的时间戳是日志**到达后端的时间**，不是日志**产生的时间**。

上一轮页面加载时设了 5 秒超时，5 秒后超时触发 DOM 兜底日志。这些日志被缓冲 500ms 后发送，HTTP 传输又有延迟。与此同时，用户可能刷新了页面，新的 Content Script 启动了。上一轮的延迟日志和这一轮的启动日志几乎同时到达后端。

**结论**：DOM 兜底并没有"太快触发"，而是上一轮的正常超时（5 秒）日志因缓冲延迟与当前轮次的启动日志叠在一起了。

### P4 根因：拦截器本身没有超时，5 秒超时是 Content Script 侧的降级机制

**设计澄清**：

```
┌─────────────────────────────────────────────────────┐
│  interceptor.js（MAIN world，常驻）                   │
│  hook fetch/XHR → 永不停止，持续监听所有请求           │
│  捕获到目标 API → postMessage → Content Script       │
└─────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────┐
│  Content Script（isolated world）                    │
│  设 5 秒超时定时器                                    │
│  ├─ 5 秒内收到 API 数据 → clearApiTimeout() → 用 API │
│  └─ 5 秒后未收到 → 触发 DOM 兜底（但拦截器仍在运行）   │
└─────────────────────────────────────────────────────┘
```

- **拦截器没有超时**：`interceptor.js` hook 了 `fetch/XHR`，只要页面有请求就会拦截，永不停止。
- **5 秒超时是 Content Script 的降级决策**：如果 5 秒内没收到拦截器的数据，用 DOM 兜底提供基本数据（无薪资）。
- **超时后拦截器仍运行**：后续如果捕获到 API 数据（如滚动加载），会通过 `sentJobTracker` 去重后发送。
- **代价**：如果首屏 API 数据在 DOM 兜底之后到达，API 数据会被去重跳过（薪资信息丢失），但这是可接受的降级代价。

### P5 根因：`handleJobsExtracted` 缺少 `await initSession()`

**数据流**：
```
用户滚动页面
  ↓
DOM observer 捕获新岗位
  ↓
Content Script 发送 JOBS_EXTRACTED 消息
  ↓
SW 唤醒（可能被 Chrome 回收过，accessToken = null）
  ↓
handleJobsExtracted 检查 getAccessToken() → null
  ↓
返回 { ok: false, error: '未登录' }
  或者 fetchBackend 不带 Authorization 头 → 后端返回 401
```

**根因**：MV3 的 SW 空闲 30 秒后会被 Chrome 回收。SW 重启时 `accessToken` 丢失（只存在内存中）。`handleJobsExtracted` 没有像 `handleLog` 那样先 `await initSession()` 恢复 token，直接检查 `getAccessToken()` 得到 null。

`fetchBackend` 的 401 自动刷新逻辑条件是 `resp.status === 401 && accessToken`：
- `accessToken` 为 null 时，请求不带 `Authorization` 头
- 后端返回 401
- `401 && null` → false → 不触发 refresh → 直接抛 `BackendError(401)`

**修复**：在 `handleJobsExtracted` 开头加 `await initSession()`。

## 三、修复方案

### 修复 F1：`handleJobsExtracted` 加 `await initSession()`（修复 P5）

**文件**：`extension/src/background/router.ts`

```typescript
async function handleJobsExtracted(...): Promise<ChromeMessageResponse> {
  // ...
  // 等待 SW 启动时的静默刷新完成，避免 SW 被 Chrome 回收后 token 丢失导致 401
  await initSession()

  if (!getAccessToken()) {
    return { ok: false, error: '未登录，请先在 Popup 登录' }
  }
  // ...
}
```

### 修复 F2：消息监听器改为 origin 校验（修复 P1，已在上轮完成）

**文件**：`extension/src/content/content.ts`

```typescript
// 修复前：
if (event.source !== window) return  // ← 主世界 window ≠ isolated world window，永远 true

// 修复后：
if (event.origin !== window.location.origin) return  // 只校验来源 origin
```

### 修复 F3：interceptor.js 修复 TDZ + targetOrigin（已在上轮完成）

**文件**：`extension/public/interceptor.js`

- 将 `const TARGET_API_PATTERNS` / `MESSAGE_TYPE` / `LOG_MESSAGE_TYPE` 声明移到 `sendLog()` 首次调用之前
- `postMessage` 的 `targetOrigin` 从 `window.location.origin` 改为 `'*'`
- 添加 `DEBUG_ALL_REQUESTS = true` 诊断日志

### 修复 F4：API 超时设计澄清（无代码变更，仅注释更新）

**文件**：`extension/src/content/content.ts`

在 `setupApiTimeout()` 的注释中明确说明：
- 拦截器本身是常驻监听的，不会因超时而停止
- 5 秒超时是 Content Script 侧的首屏降级机制
- 超时后拦截器仍运行，后续捕获的 API 数据通过去重处理

## 五、技术复盘

### 核心逻辑

**API 拦截器架构**：

```
┌─ SW (service_worker.ts) ──────────────────────────────┐
│  registerContentScripts(world: 'MAIN', document_start) │
│  → Chrome 在页面 JS 之前同步注入 interceptor.js         │
└────────────────────────────────────────────────────────┘
                        ↓
┌─ MAIN world (interceptor.js) ─────────────────────────┐
│  hook window.fetch → 包装函数 → 检查 URL 是否匹配       │
│  hook XMLHttpRequest.prototype.open/send → 同上        │
│  匹配 → clone response → parse JSON → postMessage      │
│  （常驻监听，永不停止）                                  │
└────────────────────────────────────────────────────────┘
                        ↓ window.postMessage
┌─ isolated world (content.ts) ─────────────────────────┐
│  message 监听器 → 校验 origin → 解析数据 → 去重         │
│  → sendMessageToBackground → SW → POST /api/jobs/     │
│  5 秒超时 → DOM 兜底（降级）                            │
└────────────────────────────────────────────────────────┘
```

### 关键技术点

1. **Chrome MV3 多世界模型**：MAIN world 和 isolated world 共享 DOM 但有独立的 JavaScript 堆。`window` 对象不是同一个引用，`postMessage` 的 `event.source` 指向发送方的 `window`。

2. **TDZ（Temporal Dead Zone）**：`const` 声明会被提升但不会初始化。在声明语句执行前访问它会抛 `ReferenceError`。函数声明会整体提升，但如果函数体内访问了 TDZ 中的 `const`，调用时仍会报错。

3. **`postMessage` targetOrigin**：
   - `'*'`：不检查接收方 origin，任何窗口都能收到
   - `window.location.origin`：只允许同源窗口接收
   - 在跨世界场景中，`'*'` 更安全（不会因 origin 检查失败而丢消息）

4. **MV3 SW 生命周期**：SW 空闲 30 秒后被 Chrome 回收。模块级变量（`accessToken`）在 SW 重启时丢失。所有 handler 都应先 `await initSession()` 确保 token 已恢复。

5. **日志缓冲延迟**：`remoteLog` 的 500ms 批量缓冲会导致日志时序错乱。后端终端的时间戳是日志到达时间，不是产生时间。分析时序问题时不能依赖终端时间戳。

### 潜在风险

1. **API URL 不匹配**：Boss 可能修改 API 路径。`DEBUG_ALL_REQUESTS` 日志能帮助快速定位。
2. **SW 注册失败**：如果 Chrome 版本不支持 `world: 'MAIN'`，只能靠 fallback 路径（有竞态）。
3. **`initSession` 阻塞**：所有 handler 都 `await initSession()` 会增加首次请求延迟（约 200-500ms）。但这是必要的，否则会 401。

## 六、面试题沉淀

### 题目 1：Chrome 扩展 MV3 中，主世界脚本通过 `window.postMessage` 发送消息，Content Script 的 `message` 监听器里 `event.source === window` 为什么永远为 false？

**考察点**：Chrome 扩展多世界模型、JavaScript 引用相等性

**标准答案**：
Chrome MV3 的多世界模型中，MAIN world 和 isolated world 各自有独立的 JavaScript 堆。虽然它们共享同一个 DOM，但 `window` 对象是两个不同的 JavaScript 对象。`postMessage` 的 `event.source` 指向发送方所在世界的 `window`，而 Content Script 里的 `window` 是 isolated world 的 `window`，两者引用不相等。正确的做法是只校验 `event.origin`，不比较 `window` 引用。

### 题目 2：JavaScript 中 `function foo() { console.log(x) }` 被调用时，如果 `const x = 1` 在调用之后声明，会发生什么？

**考察点**：TDZ（Temporal Dead Zone）、变量提升

**标准答案**：
`function foo` 是函数声明，会整体提升到作用域顶部，所以可以在声明前调用。但 `const x` 虽然会被提升，却不会初始化——它处于 TDZ 中。调用 `foo()` 时，函数体内访问 `x` 会抛 `ReferenceError: Cannot access 'x' before initialization`。如果这个调用被 try/catch 包裹，错误会被静默吞掉，导致难以排查。`let` 也有同样的 TDZ 行为，`var` 则不会（会返回 `undefined`）。

### 题目 3：MV3 扩展的 Service Worker 被回收后，内存中的 access_token 丢失，如何避免后续请求 401？

**考察点**：MV3 SW 生命周期、Token 刷新机制

**标准答案**：
SW 空闲 30 秒后会被 Chrome 回收，模块级变量全部丢失。恢复方案：
1. 登录时后端设置 HttpOnly Cookie 保存 refresh_token
2. `backendUrl` 持久化到 `chrome.storage.local`
3. SW 重启时调用 `initSession()`：从 storage 读取 `backendUrl`，用 Cookie 中的 refresh_token 调用 `/api/auth/refresh` 换发新的 access_token
4. 所有 handler 在处理请求前先 `await initSession()`，确保 token 已恢复
5. `initSession()` 内部用 Promise 去重，避免并发调用多次刷新
