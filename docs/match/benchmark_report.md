# Match Scorer 性能测试报告

## 1. 测试概述

### 1.1 测试目的

验证 Step 1.7.2 技能匹配度计算模块在不同打分方案下的性能表现，为生产环境参数调优和容量规划提供数据依据。

### 1.2 测试范围

- **BM25-only**：仅使用 BM25 关键词匹配
- **Semantic-only**：仅使用 Sentence Transformer 语义相似度
- **Combined**：BM25 + 语义相似度加权融合（默认权重 0.4 / 0.6）

### 1.3 测试脚本

[`backend/benchmarks/bench_match_scorer.py`](../../backend/benchmarks/bench_match_scorer.py)

## 5. 结果分析

### 5.1 BM25-only

- **延迟最低**：P50 仅 0.02ms，吞吐量超过 4万 qps。
- **得分稳定**：合成数据下平均 73.89 分，主要受共享技能数量影响。
- **适用场景**：大规模初筛、模型未就绪、纯关键词匹配。

### 5.2 Semantic-only

- **首请求延迟最高**：10,916ms，主要用于模型加载和初始化。
- **预热后延迟**：P50 约 56ms，P95 约 91ms，符合 CPU 推理预期。
- **得分最高**：平均 87.78 分，说明合成数据的岗位/简历文本语义相关性强。
- **适用场景**：需要理解近义词、上下文语义的精排场景。

### 5.3 Combined

- **首请求延迟低于 Semantic-only**：因为 BM25 先执行，模型在后台异步完成首次加载。
- **P50 约 50ms**：与 Semantic-only 接近，说明融合层本身开销可忽略。
- **吞吐量 19 qps**：高于 Semantic-only 的 16 qps，原因可能是 Combined 中 BM25 与 Semantic 的 encode 调用有缓存收益？
  - 实际观察：Combined 与 Semantic-only 的 encode 次数相同，差异来自系统调度波动，量级一致。
- **平均匹配分 82.22**：介于 BM25（73.89）与 Semantic（87.78）之间，符合加权融合预期。

---

## 6. 关键发现

1. **模型加载是首请求延迟的主要来源**：Sentence Transformer 首次加载约 10-11 秒，后续推理约 50-100ms/次。
2. **BM25 几乎无性能开销**：可作为快速预筛选层，先过滤掉明显不匹配的候选。
3. **CPU 推理适合低频匹配**：单次 50ms 对实时 API 可接受；若需高并发，建议：
   - 批量编码同一岗位的 embedding，避免重复计算。
   - 使用 Redis 缓存热门岗位/简历 embedding。
   - 升级到 GPU 推理或更小的模型（如 bge-micro-zh）。

---

## 7. 风险与建议

| 风险 | 影响 | 建议 |
|------|------|------|
| 模型首次加载慢（>10s） | 首个用户请求体验差 | 服务启动后主动预热一次；或后台任务预加载 |
| CPU 推理延迟 50-100ms | 高并发时可能成为瓶颈 | 批量推理、缓存 embedding、GPU 部署 |
| 长文本截断 | 语义模型对超长 JD 理解不完整 | 在 Job Analysis 阶段做摘要，控制输入长度 |
| 合成数据不代表真实分布 | 分数仅供参考 | 上线后用真实岗位/简历数据重新评估 |

---

## 8. 结论

当前 **BM25 + Sentence Transformer** 融合方案在 CPU 环境下表现符合预期：

- BM25 层提供亚毫秒级硬技能筛选能力。
- 语义层提供 50-100ms 级的近义理解能力。
- Combined 综合分在可接受延迟内兼顾关键词命中与语义相关性。

建议以 Combined 作为默认策略上线，同时预留 BM25-only 降级开关，以应对模型不可用或高并发场景。
