# AI Career Copilot 浏览器插件 UI 布局与样式设计文档

- **版本**：v0.1
- **日期**：2026-07-05
- **适用范围**：Chrome / Edge Manifest V3 插件（`extension/`）
- **核心形态**：SidePanel 侧边栏常驻 + Popup 入口/登录

---

## 1. 设计目标

1. **以岗位海投为主线**：所有 UI 围绕「发现岗位 → 分析匹配 → 生成沟通 → 投递跟进」展开，不堆无关功能。
2. **状态透明**：用户随时能看到系统正在做什么、哪一步卡住、是否需要介入。
3. **少 AI 味**：不用机器人头像、炫光渐变、满天星的装饰；用清晰的信息层级和克制的色彩，更像一个专业的生产力工具。
4. **常驻不打扰**：SidePanel 可折叠/展开，不遮挡 Boss 直聘主页面核心内容。

- **Popup（点击扩展图标）**：只负责登录、打开 SidePanel、登出、显示后端状态。宽 320px，生命周期短，不放复杂操作。

- **Service Worker**：做 token 中转、任务轮询、WebSocket/长连接管理。

## 4. SidePanel 布局详述

### 4.1 整体尺寸

- **宽度**：默认 380px，最小 320px，最大 480px。
- **高度**：100vh，内部 flex 列布局。
- **背景**：`#F8FAFC`（slate-50），与白色卡片形成层级。

### 4.2 Header（固定 48px）

```
┌──────────────────────────────────────┐
│ AI Career Copilot   海投助手   ●  12岗位 │
└──────────────────────────────────────┘
```

- 左侧：产品名 + 副标题「海投助手」。
- 右侧：
  - 后端状态指示灯（直径 8px 圆点）：
    - 绿色 `#22C55E`：在线
    - 红色 `#EF4444`：离线
    - 灰色 `#9CA3AF`：检测中
  - 当前页岗位数量徽章。

### 4.3 全局任务状态条（核心）

位于 Header 下方，高度可折叠，默认 36px，展开后显示多线程详情。

```
┌──────────────────────────────────────┐
│ ▶ 执行中：分析 3 个岗位，2 个 HR 对话待回复  │
│   最后心跳：08:42:15                   │
└──────────────────────────────────────┘
```

- 常态显示：当前主任务名 + 计数 + 最后心跳时间。
- 如果某个 LangGraph thread 超过 30 秒没有心跳，状态条变红并提示「某线程可能卡住」。
- 点击状态条展开「任务抽屉」，列出每个线程：
  - 线程名称（如「HR-张三-Java 后端」）
  - 当前步骤（等待回复 / 生成话术 / 等待用户确认）
  - 状态图标：进行中 / 成功 / 失败 / 卡住
  - 操作：重试 / 暂停 / 查看日志

### 4.4 Tab 导航

```
┌───────┬───────┬───────┬───────┐
│  岗位  │  沟通  │  简历  │  设置  │
└───────┴───────┴───────┴───────┘
```

- 胶囊式下划线激活态，使用主色 `#4F46E5`（indigo-600）。
- Tab 上可带小红点徽章，表示有需要用户处理的事件（如 AI 需要你确认某条回复）。

### 4.5 内容区

#### 岗位 Tab

左右分栏：

```
┌──────────────────┬─────────────────────┐
│ 岗位列表 (40%)    │  岗位详情 (60%)      │

│ • 全栈工程师      │  标签/匹配度         │

│                  │  [生成沟通话术]      │
```

- 列表项显示：标题、公司、薪资、匹配度分数、创建状态（已创建 / 失败 / 待创建）。
- 详情区：标题、公司、地点、薪资、标签、原始 JD 摘要、AI 分析结果（难度/匹配度/缺失技能）。

#### 沟通 Tab

核心区域，三部分：

```
┌──────────────────────────────────────┐
│ 活跃线程列表（可折叠）                 │
├──────────────────────────────────────┤
│                                      │
│           聊天记录区                  │
│                                      │
├──────────────────────────────────────┤
│ 快捷操作 │ 输入框          │ 发送    │
└──────────────────────────────────────┘
```

- **活跃线程列表**：每个 HR 对话一个卡片，显示最新状态、是否需要用户介入。
- **聊天记录区**：
  - 用户消息靠右，浅灰背景。
  - AI 消息靠左，白色背景。
  - AI 需要用户介入时，显示「等待你确认」高亮卡片，带「同意 / 修改 / 跳过」按钮。
- **快捷操作**：
  - 「帮我看看这个岗位」
  - 「生成打招呼话术」
  - 「暂停所有自动回复」

#### 简历 Tab

```
┌──────────────────────────────────────┐
│ [拖拽上传简历] 或 [选择文件]           │
├──────────────────────────────────────┤
│ 当前活跃简历                          │
│ 姓名/电话/邮箱/期望岗位/工作年限       │
├──────────────────────────────────────┤
│ 历史简历列表                          │
└──────────────────────────────────────┘
```

- 上传区：支持 PDF/DOCX，大小上限 10MB，拖拽高亮边框。
- 上传后显示解析进度：解析中 → 成功 / 失败。
- 活跃简历用蓝色边框高亮，可一键切换。

#### 设置 Tab

分两个卡片：

**API 设置**

- 后端地址（Backend URL）：默认 `http://localhost:8000`
- 登录状态 / 重新登录
- 测试连通性按钮

**模型设置**

- Provider：单 Provider 下拉（OpenAI / DeepSeek / Mimo），与后端 `LLMClient` 对齐。
- Base URL：Provider API 地址。
- API Key：密码框，存储在 Service Worker 内存或后端用户配置，不落 localStorage。
- Model：模型名称输入框，如 `deepseek-chat`。
- 温度、Max Tokens（可选，折叠在「高级」里）。
- 「测试连接」按钮，调用后端测试接口返回模型列表或简单响应。

> 说明：当前后端通过 `.env` 读取 LLM 配置。插件中的模型设置建议由后端新增「用户级模型配置」接口持久化；开发阶段可先写入后端 `.env`，再由插件读取当前生效配置。

### 4.6 Footer（固定 28px）

- 左侧：版本号。
- 右侧：当前登录用户名称。
- 中间（可选）：RabbitMQ/任务队列健康状态小图标。

## 5. 视觉样式规范

### 5.1 色彩系统

主色采用中性灰 + 靛蓝点缀，避免紫色/渐变 AI  cliché。

| Token | 色值 | 用途 |
|-------|------|------|
| `--bg-base` | `#F8FAFC` | SidePanel 背景 |
| `--bg-card` | `#FFFFFF` | 卡片、消息气泡 |
| `--border` | `#E2E8F0` | 分割线、边框 |
| `--text-primary` | `#0F172A` | 主文字 |
| `--text-secondary` | `#64748B` | 次要文字、描述 |
| `--text-tertiary` | `#94A3B8` | 占位、时间戳 |
| `--accent` | `#4F46E5` | 主按钮、链接、激活态 |
| `--accent-hover` | `#4338CA` | 按钮悬停 |
| `--success` | `#22C55E` | 成功、在线 |
| `--warning` | `#F59E0B` | 警告、等待用户 |
| `--error` | `#EF4444` | 错误、卡住 |
| `--info` | `#3B82F6` | 运行中、提示 |

### 5.2 字体

- 使用系统字体栈，保证在 Windows/macOS 上都清晰：
  `-apple-system, BlinkMacSystemFont, "Segoe UI", "PingFang SC", "Microsoft YaHei", sans-serif`
- 字号层级：
  - 标题：14px / 600
  - 正文：13px / 400
  - 辅助：12px / 400
  - 微标：10px / 500

### 5.3 间距与圆角

- 基础间距 4px，使用 4/8/12/16/20/24 递增。
- 卡片内边距：12px。
- 圆角：卡片 8px，按钮 6px，输入框 6px，标签 4px。
- 阴影：仅卡片使用 `0 1px 3px rgba(0,0,0,0.05)`，不用强阴影。

### 5.4 组件风格

- **按钮**：
  - 主按钮：实心靛蓝，白字。
  - 次按钮：白色背景 + 灰边框。
  - 危险按钮：白色背景 + 浅红边框 + 红字。
- **输入框**：1px 灰边框，focus 时 2px 靛蓝外发光（`ring-2 ring-indigo-100`），不用花哨动效。
- **卡片**：白色背景、浅灰边框、8px 圆角、12px 内边距。
- **标签**：小圆角胶囊，浅灰底，用颜色区分语义。
- **状态指示**：
  - 运行中：蓝色小圆点 + 脉冲动画。
  - 等待用户：橙色静态圆点。
  - 成功：绿色对勾。
  - 失败/卡住：红色感叹号。

### 5.5 减少 AI 味的具体做法

- 不使用机器人头像，AI 消息用简洁的「AI」文字标签。
- 不使用星空、神经网络、紫色渐变等装饰背景。
- 动效只保留必要的：加载 spinner、状态脉冲、消息进入的淡入。
- 文案避免「智能」「赋能」等套话，用「分析中」「等待回复」「需要确认」等直白状态。

## 6. 关键模块设计

### 6.1 API 设置

| 字段 | 说明 |
|------|------|
| 后端地址 | 默认 `http://localhost:8000`，登录时一并填写 |
| 登录态 | access_token 存 Service Worker 内存，refresh_token 走 HttpOnly Cookie |
| 健康检查 | 启动/手动触发 `GET /health` |
| 环境切换 | 设置页可保存多套后端地址（本地/测试/生产），但当前只能激活一个 |

### 6.2 Model 设置

与后端 `app/integrations/llm/llm_client.py` 对齐，采用单 Provider 通用配置：

| 字段 | 说明 |
|------|------|
| Provider | `mimo` / `deepseek` / `openai` |
| Base URL | Provider API 基地址，可编辑 |
| API Key | 密码框，blur 后显示 `sk-****xxxx` |
| Model | 模型名，如 `deepseek-chat`、`gpt-4o-mini` |
| 高级参数 | 温度、Max Tokens、Timeout（默认折叠） |
| 测试连接 | 调后端测试接口，返回连接状态和可用模型列表 |

### 6.3 简历传输

- 上传入口放在「简历」Tab 顶部，使用拖拽区。
- 上传后调用 `POST /api/resumes/upload`。
- 解析状态分三级：
  - 上传中：进度条。
  - 解析中： spinner + 「正在提取简历信息…」。
  - 成功：显示解析出的关键字段（姓名、电话、邮箱、工作年限、期望岗位、技能）。
  - 失败：红色提示 + 重试按钮。
- 用户可同时上传多份简历，但只能有一份「活跃简历」，投递时默认使用活跃简历。

### 6.4 我与 AI 的交流区

聊天区不仅是问答，更是人机协作入口：

- **用户消息**：显示用户输入的指令或确认。
- **AI 消息**：
  - 普通通知：分析完成、话术已生成。
  - 需要介入：高亮边框卡片，要求用户确认/补充信息，提供快捷按钮。
  - 错误/卡住：红色边框卡片，说明原因，提供重试。
- **快捷指令栏**：常驻在输入框上方，减少打字：
  - 「分析当前岗位」
  - 「生成打招呼」
  - 「暂停自动沟通」
  - 「这个岗位不匹配，跳过」
- **线程切换**：当同时与多个 HR 对话时，左侧线程列表可切换上下文。

### 6.5 AI 执行流程与卡住检测

参考你给出的 LangGraph 多线程调度方案，UI 需要把「后台执行状态」翻译为用户可见的信息：

#### 执行模型

- 每个 HR 对话对应一个 LangGraph thread，thread_id 稳定。
- 调度器同一时间最多主动发起一个对话，处理完一个再切下一个。
- 当需要等待 HR 回复或用户输入时，graph 进入 `interrupt/wait_for_input`，持久化 checkpoint。

#### UI 映射

| 后台状态 | UI 展示 | 用户感知 |
|---------|--------|---------|
| thread 正在运行 | 蓝色脉冲圆点 + 当前步骤名 | 系统正在处理 |
| thread 在等待 HR 回复 | 灰色时钟 + 「等待对方回复」 | 正常等待，无需操作 |
| thread 在等待用户输入 | 橙色高亮 + 「需要你确认」 | 必须处理 |
| thread 超过 30s 无心跳 | 红色感叹号 + 「可能卡住」 | 可重试或查看日志 |
| thread 失败 | 红色 + 错误摘要 | 可重试 |
| thread 完成 | 绿色对勾 | 已完成 |

#### 卡住判定规则

- **心跳机制**：Service Worker 每 5 秒轮询一次后端 `/api/tasks/{task_id}` 或 WebSocket 推送 `heartbeat` 事件。
- **卡住阈值**：
  - RUNNING 状态超过 30 秒无 updated_at 更新 → 提示「可能卡住」。
  - 连续 3 次轮询失败 → 提示「连接异常，请检查后端」。
- **用户操作**：
  - 「重试」：重新触发当前步骤。
  - 「跳过」：将该 thread 标记为 IDLE，调度器继续下一个。
  - 「查看日志」：展开该线程的最近 5 条执行日志。

## 7. 状态与反馈

### 7.1 全局状态

SidePanel 根组件（`extension/src/App.vue`）维护的状态机：

| 状态 | 触发条件 | UI |
|------|---------|-----|
| `loading` | 初始化中 | 居中 spinner + 「初始化中…」 |
| `not_logged_in` | 未登录 | 提示去 Popup 登录 |
| `idle` | 已登录，非 Boss 列表页 | 提示打开 Boss 职位搜索页 |
| `extracting` | 正在解析当前页岗位 | spinner + 已发现岗位数 |
| `ready` | 岗位列表已就绪 | 显示岗位 Tab 内容 |
| `running` | 有 AI 任务在执行 | 显示任务状态条 + 聊天区进度 |
| `error` | 发生错误 | 错误图标 + 信息 + 重试按钮 |

### 7.2 任务级状态

每个异步任务（岗位分析、匹配、沟通话术生成）使用后端 Task 模型状态：

`PENDING → RUNNING → COMPLETED / FAILED / CANCELLED`

UI 中用进度条或步骤器展示：

```
[ 排队 ] ── [ 分析中 ] ── [ 完成 ]
```

- 当前步骤高亮。
- 失败时步骤变红并显示错误信息。

## 8. 动画与微交互

- **加载**：统一使用 18px 圆环 spinner，颜色与当前语义一致。
- **状态脉冲**：运行中的状态圆点使用 `pulse` 动画，1.5s 循环。
- **消息进入**：聊天消息从下方淡入，200ms，缓动 `ease-out`。
- **任务抽屉展开**：高度从 0 过渡到 auto，使用 grid 动画或 max-height 技巧，250ms。
- **按钮悬停**：背景色加深，150ms 过渡，不用位移或缩放。

## 10. 实现建议

1. **先复用现有组件**：在 `extension/src/App.vue` 基础上增加 Tab 和任务状态条，不要一次性重写。
2. **状态集中管理**：SidePanel 状态继续放在 `useSidePanelStore`，新增 `taskRunner` store 管理多线程任务。
3. **消息协议扩展**：在 `chrome_message.ts` 中新增 `TASK_STATUS_UPDATED`、`AGENT_MESSAGE`、`HEARTBEAT_TIMEOUT` 等类型。
4. **卡住检测放在 Service Worker**：SW 轮询任务状态，计算心跳超时后再推送给 SidePanel，避免 UI 层直接轮询。
5. **先实现单 Provider 模型设置**：与后端 `LLMClient` 对齐，Provider 下拉框固定 `mimo/deepseek/openai`。
6. **先跑通核心闭环**：岗位提取 → 分析 → 生成话术 → 任务状态展示，再扩展聊天区和多线程调度 UI。

## 11. 设计原则速查

- 信息优先：每一个像素都为岗位投递服务。
- 状态可见：系统在哪一步、等谁、有没有卡住，一眼可见。
- 克制美学：少装饰、少颜色、少动效；靠间距和字体层级建立秩序。
- 人机协作：AI 是助手，不是主角；需要用户决策时明确打断，不需要时安静执行。
