## 6. 核心开发原则(强制遵守)

### 6.1 分层硬约束

| 层级 | 禁止 |
|------|------|
| Route | 写业务逻辑 |
| Service | 直接写 SQL |
| Repository | 依赖 HTTP |
| DTO | 泄露 ORM Model |

### 6.2 异步优先

- 所有 IO 必须 `async/await`(DB / Redis / HTTP / 文件 / MQ)
- 禁止伪异步(`async def` 内 `requests.get` / `time.sleep`)
- 必须解释 Event Loop 调度原理

### 6.3 防御性编程

- 精确异常捕获 + 结构化日志 + 保留 traceback
- 零信任:用户输入 / 第三方 API / 环境变量 / DB 脏数据全部校验
- 禁止 `except: pass`

### 6.4 第三方库优先

| 能力 | 推荐库 |
|------|--------|
| 配置 | pydantic-settings |
| HTTP 客户端 | httpx(async) |
| ORM | SQLAlchemy 2.x async |
| 重试 | tenacity |
| 限流 | aiolimiter |
| 校验 | Pydantic |
| 日志 | loguru / structlog |
| 加密 | cryptography / pyjwt |

### 6.5 Git 规范

```
feat(auth): add jwt refresh token support
fix(redis): resolve async connection leak
refactor(api): decouple service layer
docs(readme): update setup instructions
test(match): add scorer boundary cases
```

**禁止**: `update` / `fix bug` / `改了一下` / 大段无关修改塞同一 commit

### 6.6 输出质量 Checklist(逐项打勾)

- [ ] 职责单一、< 50 行?
- [ ] 完整 Type Hints + 返回值类型?
- [ ] 入参 Pydantic 校验?
- [ ] 精确异常 + 结构化日志 + traceback?
- [ ] 全部 IO 用 async/await?
- [ ] 中文注释(解释"为什么")?
- [ ] 无并发隐患(Race / 死锁 / 阻塞 Event Loop)?
- [ ] 单元测试覆盖正常/异常/边界?
- [ ] 解释底层原理(协程 / Event Loop / DI)?
- [ ] 性能分析(SQL 次数 / N+1 / 缓存命中)?

### 6.7 强制"先解释后编码"

写代码前先说清 5 件事:
1. **模块职责** — 这个模块做什么
2. **输入输出** — 数据如何进出
3. **生命周期** — 对象何时创建/销毁
4. **数据流** — 数据如何流转
5. **设计动机 + 方案权衡** — 为什么这样设计、为什么不用其他方案

### 6.8 强制技术复盘

每次输出代码后必须包含:
- **核心逻辑**: 解决什么问题 / 为什么这样设计 / 核心机制
- **关键技术点**: 协程 / 装饰器 / DI / 中间件 / Event Loop 底层
- **潜在风险**: Race / 内存泄漏 / SQL 注入 / 死锁 / 阻塞 Event Loop / 连接泄漏 / 缓存击穿,**必须说明如何防御**
