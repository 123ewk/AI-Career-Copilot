## 📋 开发规则

---

### 1. 拒绝端到端全包

> **No Monolithic Code Dumping**

| 🔴 严禁 | 🟢 必须 |
|---------|---------|
| 一次性生成完整项目 | 每次只实现一个模块 / 一个类 / 一个函数 |
| 输出超过 50 行且包含多个职责的代码 | 清晰说明文件路径 |
| 把路由、数据库、业务逻辑混在一起 | 说明模块职责 |
| 用"万能 utils.py"承载业务逻辑 | 先拆分模块再开发 |
| | 一个功能一个提交 |

**开发流程：**

```
需求分析 → 架构设计 → 模块拆分 → 编写接口 → 实现逻辑 → 编写测试 → 测试通过 → Git Commit → 技术复盘
```

---

### 2. 强制分层架构

> **Layered Architecture**

**默认项目结构：**

```
app/
├── api/              # HTTP 接口层
├── service/          # 业务逻辑层
├── repository/       # 数据访问层
├── model/            # ORM 模型
├── schema/           # DTO / Pydantic
├── core/             # 配置/日志/中间件
├── infra/            # Redis/Kafka/S3等基础设施
├── tests/
└── docs/
```

**架构原则：**

| 层级 | 约束 |
|------|------|
| Route | 不允许写业务逻辑 |
| Service | 不允许直接写 SQL |
| Repository | 不允许依赖 HTTP |
| DTO | 不允许泄露 ORM Model |
| 全局 | 禁止跨层乱调用 |

**依赖方向（禁止反向依赖）：**

```
api → service → repository → database
```

---

### 3. 工业级防御性编程

> **Production-Ready**

任何代码必须考虑：

- 异常情况 / 边界条件 / 高并发 / 非法输入
- 外部服务失败 / 网络超时 / 资源泄漏

| 🔴 禁止 | 🟢 必须 |
|---------|---------|
| `except: pass` | 精确异常捕获 |
| | 输出结构化日志 |
| | 明确错误上下文 |
| | 保留 traceback |

**示例：**

```python
try:
    result = await service.create_user(data)
except ValidationError as exc:
    logger.warning("Invalid user input", extra={"error": str(exc)})
    raise
except Exception as exc:
    logger.exception("Unexpected server error")
    raise
```

---

### 4. 强制异步 IO

> **Async IO First**

所有 IO 操作必须优先使用 async/await：数据库 / Redis / HTTP 请求 / 文件读写 / 消息队列

**必须解释：**

- 为什么需要异步
- 协程调度原理
- Event Loop 如何工作
- 为什么 async 不等于并行

**禁止伪异步：**

```python
# ❌ 在 async 函数中使用同步阻塞调用
async def fake():
    requests.get(...)
```

---

### 5. 强类型与自解释代码

> **Strong Typing**

| 🟢 必须 | 说明 |
|---------|------|
| 使用完整 Type Hints | 明确类型约束 |
| 明确返回值类型 | 不依赖隐式推断 |
| 使用 DTO/Pydantic 校验 | 入口校验 |
| 避免 Any 泛滥 | Any 是类型安全的敌人 |

**示例：**

```python
async def get_user(user_id: int) -> UserDTO:
```

**注释原则：**

| ❌ 错误 | ✅ 正确 |
|---------|---------|
| 注释"做了什么" | 注释"为什么" |
| `# 获取用户` | `# 提前校验用户状态，避免后续数据库事务无意义开启` |

---

### 6. 输入校验与零信任原则

> **Zero Trust**

**永远不要相信：** 用户输入 / 第三方 API / 环境变量 / 数据库脏数据

**必须校验：**

| 校验类型 | 说明 |
|---------|------|
| 参数校验 | 必填、格式、范围 |
| 类型校验 | Pydantic / Type Hints |
| 长度限制 | 防止超长输入攻击 |
| SQL 注入防护 | 参数化查询 |
| XSS 防护 | 输出转义 |
| 文件类型校验 | 白名单机制 |

### 7. 可观测性优先

> **Observability First**

| 🔴 禁止 | 🟢 必须 |
|---------|---------|
| `print("error")` | structured logging |
| | request_id / trace_id |
| | latency metrics |

**日志要求：**

```python
logger.error(
    "Create order failed",

        "user_id": user_id,
        "trace_id": trace_id

```

### 8. 并发安全

> **Concurrency Safety**

涉及共享资源时必须分析：

- 是否线程安全 / 协程安全
- 是否存在 Race Condition
- 是否需要锁 / 是否会死锁

**必须解释：**

- asyncio.Lock 原理
- GIL 的影响
- 为什么 async 下依然可能数据竞争

### 9. 性能预算意识

> **Performance Budget**

每个核心模块必须分析：

| 分析维度 | 说明 |
|---------|------|
| 时间复杂度 | 算法效率 |
| 空间复杂度 | 内存占用 |
| IO 次数 | 网络开销 |
| SQL 次数 | 数据库压力 |
| N+1 Query | 关联查询陷阱 |
| 缓存需求 | 是否需要缓存 |

**必须说明当前瓶颈：**

```
当前瓶颈：
- 数据库查询过多
- Redis 命中率低
- 外部 API 延迟高
```

### 12. 依赖管理与第三方库优先

> **Dependency First, No Re-inventing the Wheel**

工程中**能使用成熟第三方库的能力，禁止自研**。自研前必须先调研生态，否则视为重复造轮子。

| 🔴 禁止 | 🟢 必须 |
|---------|---------|
| 自研 JSON 解析、HTTP 客户端、ORM | 优先使用 Pydantic、httpx、SQLAlchemy 等成熟库 |
| 自研配置加载器（YAML/ENV 解析） | 使用 pydantic-settings、dynaconf 等 |
| 自研重试 / 限流 / 熔断 | 使用 tenacity、aiohttp 的 retry 中间件、circuitbreaker 等 |
| 自研加密 / 哈希 / 签名 | 使用 cryptography、pyjwt、passlib 等经过审计的库 |
| 自研异步任务调度 | 使用 Celery / APScheduler / Arq 等 |
| 自研序列化 / 协议编解码 | 使用 msgspec、protobuf、orjson 等 |

**依赖引入流程：**

```
调研社区方案 → 选择维护活跃的库（最近 6 个月有提交）→ 评估 License / 安全 CVE
            → 锁定版本（>=x.y.z,<x+1）→ 在 PR 中说明引入理由
```

| 约束 | 说明 |
|------|------|
| 锁版本 | `pyproject.toml` / `requirements.txt` 必须固定版本，禁止 `*` / 不指定版本 |
| 区分类型 | `dependencies`（运行时）vs `[project.optional-dependencies]`（dev/test/docs） |
| 引入理由 | 每个新增依赖必须在 PR/Commit 中说明：解决了什么、为什么选它、trade-off 是什么 |
| 安全审计 | 引入前检查 `pip-audit` / `safety` 是否报 CVE |
| 依赖精简 | 禁止"为了用一个小功能引入整个框架"，必要时只引子模块 |

**什么时候允许自研？**

- 现有方案都不满足业务约束（性能 / 合规 / 部署环境）
- 自研复杂度远低于接入第三方库的复杂度
- 必须有「为什么不用现有库」的明确说明

### 16. 强制"先解释后编码"

写代码前必须先说明：

1. **模块职责** — 这个模块做什么
2. **输入输出** — 数据如何进出

> ⚠️ 只有确认后再开始编码

| 缓存击穿 | 高并发穿透 |
| 高并发风险 | 资源耗尽 |

> 必须说明如何防御

## 🔥 21. 问题复盘与事故沉淀机制

> **Issue RCA & Engineering Postmortem**

当开发过程中遇到以下问题时，必须生成复盘文档：

| 类别 | 触发场景 |
|------|---------|
| 🐛 代码问题 | 严重 Bug / 并发问题 / 内存泄漏 / 数据错乱 / 死锁 |
| 🌐 网络问题 | WebSocket 异常 / 异步阻塞 / 第三方服务问题 |
| 🤖 AI 问题 | Agent 失控 / Prompt 设计失败 / Token 爆炸 |
| 🏗️ 架构问题 | 架构问题 / 性能瓶颈 |
| 🗄️ 数据问题 | 数据库问题 |
| 🚀 运维问题 | 线上故障 / CI/CD 失败 / 部署事故 / 安全问题 |

**文档路径：** `docs/issues/`

**命名规范：**

```
2026-05-31-websocket-connection-leak.md
2026-05-31-agent-token-explosion.md
2026-05-31-async-blocking-event-loop.md
```

---

### 复盘文档必须包含以下 10 个部分：

#### 1️⃣ 问题背景（Background）

说明：问题出现在哪个模块 / 当时正在实现什么功能 / 系统架构是什么 / 涉及哪些技术栈

> **示例：** 在实现 Agent WebSocket 实时推流时，出现大量连接未释放问题，导致内存持续增长。

#### 2️⃣ 问题现象（Symptoms）

详细记录：报错信息 / 日志 / 请求链路 / 复现步骤 / 出现概率 / 影响范围

> ⚠️ 必须保留原始错误信息
>
> | 🔴 禁止 | 🟢 必须 |
> |---------|---------|
> | "报错了" | `RuntimeError: Task attached to a different loop` |
>
> 并解释：为什么这个错误危险 / 为什么容易误导

#### 3️⃣ 根因分析（Root Cause Analysis）

必须深入分析：为什么会出现问题 / 真正根因是什么 / 表面问题是什么 / 底层机制是什么

**分析角度：** 运行时机制 / 源码调用链 / Event Loop / 线程 / 内存

> ⚠️ **必须区分：现象 ≠ 根因**
>
> - 现象：WebSocket 断连
> - 根因：协程异常未正确取消，导致 connection manager 持有失效引用

#### 4️⃣ 排查过程（Debug Journey）

记录：做过哪些尝试 / 为什么失败 / 如何一步步定位 / 用了哪些工具

**工具包括：** logging / tracing / profiler / asyncio debug / memory snapshot / SQL explain / Chrome DevTools

> 必须说明：为什么这个方向错了？为什么后来换方向？

#### 5️⃣ 技术选型分析（Technical Decision）

如果问题涉及架构调整，必须说明：

| 分析维度 | 说明 |
|---------|------|
| 为什么不用方案 A | 被排除的原因 |
| 为什么选择方案 B | 选中的理由 |
| 方案之间 trade-off | 权衡分析 |

**必须分析：** 性能 / 扩展性 / 复杂度 / 可维护性 / 成本

> **示例：** 为什么不用 ThreadPool？为什么改成 asyncio.Queue？为什么不用 Redis Pub/Sub？为什么改 Kafka？

#### 6️⃣ 最终解决方案（Final Solution）

必须包含：最终架构图 / 关键代码 / 修复点 / 为什么有效

> ⚠️ 必须解释为什么这个方案真正解决了根因，而不是"改完能跑了"

#### 7️⃣ 风险与副作用（Risks & Trade-offs）

分析：是否引入新问题 / 是否影响性能 / 是否增加复杂度 / 是否影响扩展性

> 必须说明：这个修复的代价是什么？

#### 8️⃣ 如何预防（Prevention）

必须总结：

| 预防措施 | 说明 |
|---------|------|
| 如何提前发现 | 监控告警机制 |
| 如何避免再次发生 | 架构约束 |
| 应增加什么测试 | 回归测试 |
| 应增加什么监控 | 指标采集 |
| 应增加什么架构约束 | 代码规范 |

> **示例：** 增加 connection timeout / heartbeat / weakref 管理 / async task cleanup

#### 9️⃣ 学到的核心知识（Key Learnings）

必须提炼：学到了什么底层原理 / 理解了什么架构思想 / 哪些知识以前是错误理解

> **示例：** 理解了 async 并不会自动避免资源泄漏，协程对象生命周期依然需要显式管理。

#### 🔟 面试题沉淀（Interview Knowledge）

每个问题必须沉淀：

```
题目 → 考察点 → 标准答案 → 延伸问题
```

> **示例：**
>
> **Q:** 为什么 asyncio 下依然可能发生资源泄漏？
>
> **考察点：** 协程生命周期管理

---

> 💡 **核心目标：** 问题复盘的目标不是 **"记录 Bug"**，而是 **"建立工程问题分析能力"**
>
> 真正的高级工程师：不是 **"从不出问题"**，而是 **"能系统化分析并解决复杂问题"**

---

### 22. 强制「先反问后编码」

> **Clarify Before Coding — 模糊必问，不猜不默认**

这是本项目的**最高优先级行为准则**，覆盖所有 AI 输出。任何代码生成、配置变更、SQL 编写、文档产出之前，必须先通过本节过滤器。

---

#### 🚨 触发条件（满足任一必须先反问再写代码）

| # | 触发场景 | 典型示例 |
|---|---------|---------|
| 1 | **多种实现路径** | 技术选型 / 库 / 架构未指定（≥2 种合理方案并存） |
| 2 | **需求冲突** | 用户表述、既有规则、已有代码三者出现矛盾 |
| 3 | **范围不清** | 边界条件 / 异常分支 / 权限角色 / 错误码未定义 |
| 4 | **术语歧义** | "用户""会话""实时""同步"等关键概念存在多种解释 |
| 5 | **性能/安全/兼容性** | 未指定 QPS、延迟、SLA、合规要求、目标客户端 |
| 6 | **数据契约** | 入参 / 出参 / 持久化结构 / 枚举值未明确 |
| 7 | **破坏性变更** | 影响既有模块、API、数据库 Schema、依赖升级 |
| 8 | **跨模块影响** | 单点修改波及多个文件 / 服务 / 团队 |
| 9 | **可见性归属** | UI 改动的样式归属（写死 vs 主题变量）未指定 |

---

#### 🛠️ 反问规范

| 🔴 禁止 | 🟢 必须 |
|---------|---------|
| 纯文字流式提问 | 使用 **AskUserQuestion 工具**结构化反问 |
| "是 / 否"二元题 | 给出 2–4 个互斥选项 + 各自 trade-off |
| 边问边写 | 反问期间**禁止**写代码、配置、SQL、文档 |
| 隐式推荐 | 推荐项必须标注 `(推荐)` |
| 一次问超过 4 个 | 单次最多 4 个问题，避免用户疲劳 |
| 自行猜测 | 必须等待用户**显式确认**后再动手 |

**选项设计三要素：**

```
每个选项必须说明：
  1. 做什么      — 具体实现路径
  2. 不做什么    — 主动放弃的方案
  3. 代价        — 性能 / 复杂度 / 维护性 / 扩展性 trade-off
```

---

#### 🟢 例外（可静默处理，无需反问）

- 已被项目规则明确约束的行为（如必须 async/await、必须中文注释、必须 Pydantic 校验）
- 纯排版 / 注释 / 重构 / 重命名，不改变外部行为
- 已有明确模式可直接复用（项目内已有同类模块可参考）
- 用户明确说"按你判断来""先跑通再说""你定"

---

#### 📚 反问沉淀

| 沉淀动作 | 触发条件 | 落点 |
|---------|---------|------|
| 记录结论 | 任何反问结束后 | 出现在最终交付物的设计说明中 |
| 沉淀案例 | 反复出现的同类问题 | `docs/clarifications.md` |
| 升级规则 | 同一问题反问 ≥3 次 | 纳入本文件「核心原则」或新增章节 |

---

#### 🔗 与既有规则的关系（优先级链）

```
反问（最高） → 设计（核心原则 #1） → 编码（核心原则 #2-6） → 复盘（章节 21）
```

| 上游规则 | 关系 |
|---------|------|
| **核心原则 #1 先设计后编码** | 不冲突但**优先于**：先反问 → 再设计 → 再编码 |
| **AI 协作边界** | **强化**：「AI 禁止一次性生成整个系统」 |
| **强制「先解释后编码」**（章节 16） | **前置**：反问后输出的解释才算有效解释 |
| **问题复盘机制**（章节 21） | **配合**：事后发现需求理解错误必须走复盘 |

---

#### 💡 反问 vs 解释 的边界

| 维度 | 反问 | 解释 |
|------|------|------|
| 触发 | 需求**不清楚** | 需求**清楚**但实现**复杂** |
| 输出 | AskUserQuestion 选项 | 文字 + 代码 + 流程图 |
| 目的 | 防止做错 | 防止说不清 |
| 时机 | 编码**前** | 编码**后**（或编码中） |

> ⚠️ **反问不可替代解释**：反问只解决"做什么"，不解决"为什么这样做"。两者必须配合使用。
